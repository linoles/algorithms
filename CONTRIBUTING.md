# Как помочь проекту «Базовые алгоритмы C++»

Привет! 👋

Спасибо, что хотите внести вклад в этот проект. Здесь я собираю и разбираю алгоритмы и структуры данных на C++ по мере моего обучения олимпиадному программированию. Проект открыт для всех — код можно копировать, использовать в своих целях и делиться им.

**Главная цель этого репозитория** — не создать идеальную библиотеку алгоритмов, а **показать мой путь в олимпиадном программировании** и **помочь другим начинающим**: от простых решений к более сложным, с пояснениями, визуализациями и ссылками на полезные ресурсы.

Поэтому даже «неидеальный» код с комментариями «я сделал так, но можно лучше» — это ценный вклад в историю обучения.

## 🎯 Как вы можете помочь?

### 🟢 Для начинающих - отличный способ сделать первый Pull Request

- **Исправить опечатки** в README, комментариях или коде
- **Дополнить объяснения** — написать, что было непонятно вам как новичку
- **Улучшить визуализации** — добавить картинки, схемы, примеры работы
- **Добавить ссылки** на полезные материалы (статьи, видео, визуализации)

### 🟡 Для уже знающих

- **Добавить новый алгоритм** (даже если не самый оптимальный — важно показать ход мыслей)
- **Предложить оптимизацию** существующего кода с пояснением, почему так лучше
- **Написать альтернативную реализацию** того же алгоритма (например, версию с другим pivot в сортировке Хоара)

### 🔴 Для опытных

- **Добавить продвинутые структуры данных** (деревья отрезков, Фенвика, СНМ)
- **Решить интересные задачи** с подробным разбором

### 🤝 Для всех

- **Сообщать об ошибках** через [Issues](https://github.com/linoles/algorithms/issues)
- **Предлагать идеи** для новых алгоритмов или улучшений
- **Делиться ресурсами**, которые помогли вам в изучении

## 🔧 Процесс внесения изменений

Ниже описан порядок действий без конкретных команд — они зависят от вашего клиента Git (терминал, VS Code, GitHub Desktop). Смысл везде одинаков.

**1. Форк репозитория**

На странице проекта на GitHub нажмите кнопку **Fork**, чтобы создать свою копию репозитория.

**2. Клонирование**

Скопируйте *свой* форк на компьютер (ссылку даст GitHub).

**3. Синхронизация с оригиналом**

Добавьте оригинальный репозиторий как «удалённый» (обычно называют `upstream`), чтобы позже забирать свежие изменения.

**4. Создание ветки**

Название ветки должно отражать суть изменений:

- `feature/название-алгоритма` — новый алгоритм
- `fix/описание-ошибки` — исправление
- `docs/что-изменилось` — документация

**5. Внесение изменений**

- Следуйте стилю кода (см. ниже)
- **Обязательно** добавляйте комментарии
- Если алгоритм можно улучшить — напишите об этом в комментариях или пометкой `TODO`

**6. Коммит**

Формат сообщения коммита:

- `feat(область): описание` — новый алгоритм или структура данных
- `fix(область): описание` — исправление ошибки
- `docs(область): описание` — изменения в документации
- `style(область): описание` — форматирование (не влияет на логику)
- `refactor(область): описание` — улучшение кода
- `test(область): описание` — добавление тестов
- `chore(область): описание` — обслуживание

_Примеры:_
- `feat(sorts): добавить сортировку TimSort`
- `docs(ds): дополнить описание стека примером использования`
- `fix(searches): исправить выход за границы в бинарном поиске`

**7. Публикация ветки**

Отправьте (push) свою ветку в *свой* форк на GitHub.

**8. Pull Request (PR)**

На GitHub перейдите в свой форк, нажмите **Compare & pull request**. В описании PR напишите:

- Что изменилось?
- Почему это полезно?
- Если PR решает какую-то задачу, укажите `Closes #номер`

## ✅ Требования к коду

### Базовые требования

| Требование | Описание |
|------------|----------|
| **Стандарт C++** | C++17 или новее (можно C++20 с пометкой) |
| **Компиляция** | Код должен компилироваться без ошибок, как минимум, в GCC |
| **Зависимости** | Только стандартная библиотека (никаких Boost, Qt и т.д.) |

### Обработка ошибок

- Для учебных алгоритмов допустимо не обрабатывать все ошибки
- **Но** если алгоритм может упасть (деление на ноль, выход за границы) — **укажите это в комментариях!**

### Комментарии — это важно!
В этом проекте комментарии важнее идеального кода. Если вы делаете что-то не совсем оптимально, но понятно — это хорошо. Напишите, почему вы выбрали этот подход и как его можно улучшить.

## 📝 Оформление документации

### Шаблон для файла с алгоритмом
```cpp
/**
 * @file название.cpp
 * @brief Очень краткое описание (одна строка)
 * 
 * @details 
 * Что делает этот алгоритм. Как он работает. 
 * В чём его сильные и слабые стороны.
 * 
 * 📊 Сложность:
 * - Время: O(?)
 * - Память: O(?)
 * 
 * 💡 Особенности:
 * - Когда использовать
 * - На что обратить внимание
 * - Возможные улучшения
 * 
 * @param имя_параметра Описание
 * @return Что возвращает
 */

// РЕАЛИЗАЦИЯ

// TODO: оптимизировать, используя ...
// FIXME: не работает для отрицательных чисел
```

### Обновление README
Если вы добавили новый алгоритм, добавьте его в соответствующий раздел README:

```markdown
### Название алгоритма - [_/путь/к/файлу.cpp_](/тот/же/путь.cpp)

Краткое описание (1-2 предложения). Что делает, в чём особенность.

- Что принимается на вход
- Что возвращает функция
- Скорость: **O(?)**, память: **O(?)**
```

## 🖼️ Визуализация и ссылки
Этот проект уделяет особое внимание **наглядности**. Если вы добавляете алгоритм, постарайтесь добавить:

### Что можно добавить в папку assets/:

- Скриншоты работы алгоритма
- Схемы (можно нарисовать в [Excalidraw](https://excalidraw.com/), [draw.io](https://draw.io))
- Таблицы сравнения производительности

### Как добавлять ссылки в коде (в блоке @details):

```
 * 🔗 Полезные ссылки:
 * - [Визуализация на VisualGo](https://visualgo.net/...)
 * - [Видео: как работает алгоритм](https://youtu.be/...)
 * - [Статья на русском](https://habr.com/...)
В README:

💡 Больше визуализаций и примеров: (ссылка)[ссылка]
```

## 📜 Лицензия

Внося изменения в этот проект, вы соглашаетесь с тем, что ваш код будет распространяться под лицензией **MIT** (как и весь проект). Это значит, что **любой** может использовать ваш код в любых целях, включая коммерческие, с указанием авторства.
